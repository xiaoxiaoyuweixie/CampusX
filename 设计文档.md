# CampusX 校园二手交易小程序设计文档

## 第1章 引言

### 1.1 项目背景

随着高校信息化建设的不断推进，校园生活中大量交易、交流和服务行为逐渐从线下转移到线上。学生群体具有较强的二手交易需求，例如教材、考研资料、数码设备、宿舍用品、技能服务等物品或资源在毕业季、开学季和考试周期中流转频繁。传统的校园二手交易方式主要依赖微信群、QQ群、朋友圈、校园墙或线下公告栏。这类方式虽然门槛低，但也存在明显不足：信息分散、检索困难、商品状态不透明、沟通记录不统一、卖家身份缺少基础认证、管理方难以及时发现违规信息等。

CampusX 项目面向校园场景设计，目标是构建一个轻量、清晰、可管理的校园二手交易平台。系统采用微信小程序作为学生端入口，利用微信生态的低安装成本和高触达能力，使用户能够快速完成登录、浏览商品、分类筛选、发布商品、收藏商品、联系卖家和管理个人信息等操作。与此同时，系统配套设计后台 Web 管理端，供管理员查看用户、商品、分类、聊天记录和整体数据指标，并对用户与商品进行基础管理。通过小程序端与后台管理端的协同，平台能够兼顾用户体验和运营治理。

在数据存储和后端能力方面，项目使用微信云开发提供的云函数、云数据库和云存储。云函数负责承载业务逻辑，云数据库负责保存用户、商品、分类、收藏、聊天会话、聊天消息和管理员信息，云存储用于保存用户头像和商品图片。与自建服务器相比，云开发方式能够降低早期开发和部署成本，减少服务器运维压力，并且天然适配微信小程序调用链路。对于课程设计、项目原型和校园级应用的初期建设而言，这种技术选型具有较好的可落地性。

### 1.2 项目意义

CampusX 的意义主要体现在三个方面。第一，项目提升了校园二手交易的信息组织能力。通过商品分类、首页推荐、关键词搜索、详情页展示和商品状态管理，用户不再需要在大量聊天记录中查找零散信息，而是可以在结构化页面中快速了解商品标题、价格、分类、位置、图片、卖家信息和交易状态。

第二，项目提高了交易沟通的可追踪性。系统设计了消息模块，用户可以围绕具体商品发起聊天，会话与商品、买家、卖家建立关联，聊天记录统一保存到云数据库中。相比临时私聊或群内留言，这种方式更便于用户回溯交易细节，也为后续售后沟通预留了空间。

第三，项目增强了平台治理能力。后台管理系统提供用户列表、商品列表、分类列表、商品上下架、聊天记录查看和数据看板等功能。管理员能够禁用异常用户、下架不合规商品、修改分类信息，并查看平台的基础运行数据。为了保证后台操作真正生效，后端云函数会根据用户状态进行权限拦截，例如被禁用用户不能继续登录、发布商品、收藏商品或发送消息。

从工程实践角度看，本项目覆盖了移动端前端、云函数后端、云数据库设计、云存储上传、后台 Web 管理端、接口封装、权限控制和系统测试等多个环节，能够较完整地体现一个小型全栈应用从需求分析到实现落地的过程。

## 第2章 需求分析和可行性分析

### 2.1 用户角色分析

系统主要涉及三类角色：普通用户、卖家用户和后台管理员。普通用户指通过小程序登录平台的学生，主要功能包括浏览商品、搜索商品、查看分类、查看商品详情、收藏商品、联系卖家、查看消息、维护个人资料等。卖家用户本质上也是普通用户，只是在平台中额外承担商品发布和商品管理行为，可以发布商品、查看自己的发布列表、将商品标记为已售出或已下架。后台管理员通过 Web 管理系统登录，负责平台基础运营管理，包括查看用户和商品数据、禁用用户、修改商品状态、维护分类和查看聊天记录。

系统没有设计复杂的多级管理员权限，第一阶段后台管理端设置两个超级管理员账号，分别为 wu 和 xiao。管理员密码经过哈希加盐后存储在 admins 集合中，后台登录成功后通过令牌识别管理员身份。管理员只能通过后台服务访问管理接口，普通小程序用户不能直接调用后台管理能力。

### 2.2 功能需求分析

本系统的功能需求按照前端小程序端、后端云函数与数据服务端、Web 管理端三个部分展开。前端负责直接面向普通用户提供页面交互和业务入口，后端负责完成身份校验、数据读写、业务规则判断和云存储协同，Web 管理端负责面向管理员提供平台运营管理能力。三部分之间通过统一接口进行协作，共同完成校园二手交易平台的完整业务闭环。

前端小程序端主要面向学生用户，要求提供登录、首页浏览、分类查看、商品发布、商品详情、收藏、消息聊天、个人中心、设置、我的发布、我的收藏和关于我们等功能。用户进入系统后首先在登录页面输入账号和密码，账号格式限定为以 22 开头的 15 位数字，密码第一版使用默认口令 swu123456。登录成功后，前端需要保存用户基础信息并跳转至首页；登录失败时，需要向用户展示明确提示，例如账号格式错误、密码错误或账号已被禁用。

首页是小程序端的主要浏览入口，需要展示搜索栏、商品分类入口和推荐资源列表。推荐资源不应依赖用户主动搜索才显示，而应默认展示在售商品，使用户进入首页后能够直接浏览商品信息。搜索栏需要支持关键词检索，用户完成搜索后可以通过取消按钮退出搜索状态并回到首页推荐列表。分类页需要按商品分类展示资源，分类选项应包含数码电子、考研资料、书籍教材、技能服务、宿舍用品等校园常见类别。即使后端 categories 数据为空，前端也需要提供默认分类兜底，保证首页分类区域和发布页分类选择器始终可用。

商品发布与商品详情是小程序端的核心交易功能。发布页面要求用户能够填写商品标题、分类、价格、交易地点、商品描述并上传商品图片。图片上传完成后，前端获得云存储 fileID，再将 fileID 与商品表单数据一起提交给后端。商品详情页需要展示商品图片、标题、价格、分类、位置、描述、浏览量、收藏状态以及卖家头像和卖家昵称。用户可以在详情页收藏或取消收藏商品，也可以围绕商品联系卖家进入聊天页面。

个人中心相关页面需要展示用户头像、昵称、学校认证信息、个人简介、收藏数量和功能入口。设置页面要求用户可以更新头像、昵称、性别和个人简介。头像由前端先上传至云存储 avatar 文件夹，路径格式采用 avatar/{openid}-{timestamp}.xxx，上传成功后再调用用户资料接口更新数据库。我的发布页面需要按照在售、已下架、已售出三个状态展示卖家发布的商品，并提供下架或标记售出等状态操作。我的收藏页面需要展示用户收藏过的商品列表。

消息聊天页面要求用户能够围绕具体商品与卖家沟通。前端在商品详情页点击联系卖家时，需要请求后端打开或创建会话；在聊天页面中，需要展示历史消息、发送新消息、发送成功后清空输入框，并在消息列表页展示已有聊天会话。消息页面可以保留系统通知区域，但当系统通知接口不存在或没有数据时，不应影响聊天会话的展示。

后端云函数与数据服务端主要负责业务规则和数据一致性。登录云函数需要校验账号格式、默认口令、openid 绑定关系和用户状态。系统第一次识别到合法账号和对应 openid 时，需要在 users 集合中初始化用户数据，默认头像使用系统默认头像，默认昵称为“用户+6位随机数字”，性别默认为“未知”，学校默认为“西南大学”，状态默认为 enabled。之后用户再次登录时，系统根据 openid 定位用户信息，不覆盖用户已经更新过的头像、昵称、个人简介和性别等资料。如果后台将用户状态设置为 disabled，登录云函数必须拒绝该用户继续登录。

用户云函数需要支持用户资料读取、资料更新和个人数据统计。更新资料时，后端只接收前端已经上传成功的头像 fileID 以及昵称、个人简介、性别等字符串字段，并更新 users 集合中的 avatar、nickname、bio 和 gender 等字段。为了保证禁用逻辑真实有效，用户云函数还需要检查当前用户是否存在以及 status 是否为 disabled，防止被禁用用户继续修改资料或读取需要登录的个人数据。

商品云函数需要支持商品发布、商品列表、商品详情、我的发布和商品状态更新。商品发布成功后，products 集合中应新增商品记录，商品状态默认为 on_sale。商品状态在卖家侧分为三类：在售、已下架、已售出。用户不想继续售卖商品时，可以将商品状态调整为 off_shelf；商品交易完成后，可以将商品状态调整为 sold。系统当前不提供物理删除商品功能，删除类操作应理解为将商品状态调整为下架，从而保留历史数据。商品详情接口需要根据商品 openid 读取最新卖家资料，避免详情页展示过期头像。

分类、收藏和消息云函数需要分别保证对应业务数据的稳定读写。分类云函数读取 categories 集合并按 sort 排序返回启用分类，分类字段包括 categoryId、name、icon、description、sort、status 和 productCount。收藏云函数需要保证同一用户对同一商品只产生一条有效收藏记录，并同步维护 products.favoriteCount 字段，便于列表页和详情页直接展示收藏数量。消息云函数需要维护 chat_sessions 和 chat_messages 两个集合，前者保存会话摘要、参与者、商品快照、最后一条消息和未读数量，后者保存每条聊天消息的具体内容。发送消息后，后端需要同步写入消息记录并更新会话 lastMessage、lastMessageAt 和 unreadCount。

Web 管理端主要面向平台管理员，要求提供本地开发可运行的后台管理系统。管理员通过账号密码登录后，可以查看数据看板、用户列表、商品列表、分类列表和聊天记录。数据看板需要展示用户总数、禁用用户数、商品总数、在售商品数、已下架商品数、已售出商品数、分类数、会话数和消息数，帮助管理员快速了解平台运行情况。

Web 管理端的用户管理功能需要支持查看用户资料、按状态识别禁用用户，并提供禁用和启用用户操作。商品管理功能需要支持查看商品列表和调整商品状态，例如将违规商品下架。分类管理功能需要支持查看和修改分类基础信息。聊天记录查看功能需要支持管理员查看会话和消息内容，用于平台运营和异常问题排查。后台管理端不直接连接云数据库，而是通过 HTTP 访问 adminService 云函数，由云函数统一完成管理员身份验证、数据库访问和管理结果返回。

### 2.3 非功能需求分析

系统需要满足易用性、可靠性、可维护性和安全性要求。前端易用性方面，小程序页面应尽量保持操作路径短，首页进入后直接展示商品，发布页面必须保证分类可选，搜索后可以通过取消按钮回到推荐商品列表。设置、收藏、我的发布和聊天等高频功能应保持入口清晰、反馈明确，避免用户完成一次操作后不知道当前状态是否已经更新。

后端可靠性方面，云函数需要返回统一结构，前端根据 code、message 和 data 判断调用结果，避免不同模块之间错误处理不一致。涉及多集合联动的业务，例如收藏数量维护、聊天消息发送和会话摘要更新，需要尽量集中在云函数中完成，避免前端分别操作多个集合造成数据不一致。对于用户不存在、商品不存在、无权限、账号被禁用等异常场景，后端应返回明确错误信息。

系统可维护性方面，项目将不同业务拆分为 login、user、category、product、favorite、chatService 和 adminService 等云函数，避免所有逻辑集中在单个云函数中导致后期难以修改。前端通过统一 api 封装调用云函数，页面层主要负责展示和交互；Web 管理端通过统一 adminService 访问管理接口，避免直接暴露数据库访问逻辑。这样的结构有利于后续按模块继续扩展订单、评价、举报和审核等功能。

安全性方面，账号格式、默认口令、openid 绑定、管理员令牌、用户禁用状态和商品所有权校验都需要在云函数中实现，不能只依赖前端页面控制。后台管理端的管理员登录应使用 token 验证身份，管理员密码不应明文保存。普通用户被禁用后，登录、资料更新、商品发布、收藏和聊天等需要登录的操作都应在后端被拦截，保证后台管理操作能够真正影响小程序端权限。

### 2.4 可行性分析

技术可行性方面，微信小程序提供 WXML、WXSS 和 JavaScript 组件化开发能力，适合实现移动端页面和交互；微信云开发提供云函数、云数据库和云存储，能够满足用户、商品、收藏和聊天数据的存储需求；后台 Web 端采用 Vite 和原生 JavaScript 实现，开发成本较低，能够快速搭建管理界面。项目已经具备前端页面、云函数目录和后台管理端目录，技术路线清晰。

经济可行性方面，项目在初期不需要购买独立服务器，云开发环境可以支撑课程设计或小规模校园试用。后台管理系统先采用本地开发模式，不涉及复杂部署成本。后续如果用户规模扩大，可以再考虑云函数资源扩容、数据库索引优化和后台正式部署。

操作可行性方面，学生用户只需要通过微信小程序使用系统，无需额外安装 App；管理员通过浏览器访问后台管理端，登录后即可完成数据查看和管理操作。系统的主要功能与校园二手交易的自然流程一致，用户学习成本较低。

风险方面，当前系统仍处于迭代阶段，部分协议文本、系统通知、订单闭环和支付能力尚未实现；云数据库权限和云函数部署依赖微信开发者工具与云开发控制台配置；后台 HTTP 访问还需要正确配置云函数访问路径和跨域设置。这些风险可以通过阶段化开发、明确部署流程和补充测试用例逐步降低。

## 第3章 系统详细分析和设计

### 3.1 系统总体流程

CampusX 的核心业务流程可以概括为“用户登录—浏览或发布商品—收藏或联系卖家—交易状态维护—后台管理监督”。用户首次进入小程序时先进入登录页面，输入合法账号和默认口令后调用 login 云函数。云函数验证账号格式、口令和 openid 绑定关系。如果该 openid 对应用户不存在，则初始化用户记录；如果用户已存在，则读取用户资料并更新最后登录时间；如果用户已被禁用，则返回错误并拒绝登录。

登录成功后，用户进入首页。首页加载分类信息和推荐商品，推荐商品来自 products 集合中的在售商品。用户可以通过分类入口进入分类页，也可以使用搜索栏按关键词查找商品。点击商品卡片后进入详情页，详情页加载商品详情和卖家信息，并检查当前用户是否已收藏该商品。用户可以收藏商品，也可以点击联系卖家。联系卖家时 chatService 云函数会判断商品是否允许发起会话、当前用户是否为商品卖家、是否已有会话，并最终返回会话信息，前端跳转到聊天页面。

如果用户是卖家，则可以进入发布页面填写商品信息并上传图片。发布成功后 product 云函数创建商品记录，商品状态默认为 on_sale。之后卖家可以在“我的发布”中查看自己发布的商品，按在售、已下架和已售出三个状态筛选。卖家可以将商品下架，也可以标记为已售出。

后台管理员通过 admin-web 登录管理系统。后台调用 adminService 云函数完成管理员登录、数据看板加载、用户列表加载、商品列表加载、分类维护和聊天记录查看。管理员禁用用户后，users 集合中该用户 status 字段变为 disabled。为了保证禁用效果，login、user、product、favorite 和 chatService 等云函数都会检查用户状态，阻止被禁用用户继续使用核心功能。

### 3.2 登录与用户初始化流程

登录模块的输入包括账号和密码。账号必须满足“以 22 开头的 15 位数字”这一规则，密码第一版采用固定口令 swu123456，不在数据库中保存明文密码。云函数首先读取微信上下文中的 OPENID，用于识别当前微信用户。之后检查账号格式和密码是否正确，任一条件不满足都返回登录失败。

账号合法后，系统根据 openid 查询 users 集合。如果找到用户记录，则说明当前微信用户已经完成初始化或曾经登录过。此时云函数继续检查账号绑定关系，若当前 openid 已绑定其他账号，则拒绝登录；若用户状态为 disabled，则拒绝登录；否则更新 lastLoginAt 并返回用户资料。

如果 openid 没有对应用户，云函数会继续检查账号是否已被其他 openid 绑定。账号和 openid 均要求唯一，因此一个账号不能被多个微信用户重复绑定。确认未被绑定后，云函数创建新用户记录，包括 account、openid、userId、avatar、nickname、bio、gender、school、status、verified、createdAt、updatedAt 和 lastLoginAt 等字段。默认头像使用系统默认头像 fileID，默认昵称为“用户+随机数字”，gender 默认为“未知”，status 默认为 enabled。

### 3.3 商品发布与商品状态流程

商品发布流程由前端发布页面和 product 云函数共同完成。用户在发布页面选择分类、输入标题、价格、交易地点和描述，并选择商品图片。图片上传到云存储后形成 fileID 数组，随后前端调用 createProduct。云函数检查用户是否登录、用户是否存在、用户是否被禁用，再校验商品输入内容。校验通过后，系统生成 productId，写入 products 集合。

商品状态设计重点体现卖家侧管理需求。商品发布成功后进入 on_sale，即在售状态。若卖家暂时不想出售，可以在“我的发布”中执行下架操作，商品状态变为 off_shelf。若商品已经交易完成，卖家可以标记为 sold。当前阶段不提供物理删除商品功能，避免误删导致交易记录、收藏记录或聊天记录无法追踪。

商品列表和详情页只展示满足条件的商品。首页推荐和普通列表默认展示 on_sale 商品；详情页可以根据 productId 或数据库 _id 查询商品，并同时加载卖家信息。为了保证卖家头像更新后详情页展示最新头像，详情页查询商品时会根据商品 openid 读取 users 集合中的卖家资料，而不是只依赖商品创建时保存的 sellerAvatar 快照。

### 3.4 收藏流程

收藏模块围绕商品详情页和“我的收藏”页面展开。用户进入详情页时，前端调用 checkFavorite 判断当前商品是否已被收藏。点击收藏按钮后，如果未收藏则调用 addFavorite，如果已收藏则调用 removeFavorite。favorite 云函数先校验 openid、用户存在性和禁用状态，再根据 productId 查找商品记录。

为了避免重复收藏，addFavorite 会先检查 favorites 集合中是否已经存在相同 openid 和 productId 的记录。如果不存在，则新增收藏记录，并对 products 集合中对应商品的 favoriteCount 执行递增。取消收藏时，云函数找到收藏记录并删除或更新，同时减少商品 favoriteCount。列表页读取收藏记录后，再关联商品信息，形成用户收藏商品列表。“我的”页面中的收藏量可以通过用户看板接口或收藏集合统计获得。

### 3.5 消息与聊天流程

消息模块以商品为上下文建立会话。用户在商品详情页点击联系卖家时，前端调用 chatService 的 openSession。云函数读取当前 openid、查询商品、判断商品状态。只有 on_sale 和 sold 状态允许发起新会话，off_shelf 商品不允许新建会话。若买家和卖家已经存在同一商品对应会话，则直接返回已有会话，避免重复创建。

chat_sessions 集合保存会话级信息，例如 sessionId、productId、participants、buyerOpenid、sellerOpenid、productSnapshot、buyerSnapshot、sellerSnapshot、lastMessage、lastMessageAt、unreadCount、status、createdAt 和 updatedAt。chat_messages 集合保存消息级信息，例如 messageId、sessionId、senderOpenid、receiverOpenid、content、type、status、createdAt 等。发送消息时，云函数先检查用户状态、会话是否存在、当前用户是否为参与者，再写入 chat_messages，并同步更新 chat_sessions 的 lastMessage 和 unreadCount。

前端聊天页面加载消息列表后，用户输入内容并点击发送。发送成功后输入框应清空，消息列表自动刷新或追加新消息。消息页面加载会话列表时，如果系统通知接口不可用或没有数据，不影响聊天会话区域展示。该设计保证了消息模块的核心功能与扩展通知功能相互独立。

### 3.6 后台管理流程

后台管理系统由 admin-web 前端和 adminService 云函数组成。管理员在浏览器中打开后台页面，输入账号和密码登录。adminService 查询 admins 集合，验证管理员状态、密码哈希和角色。登录成功后返回 token，前端保存在本地状态中，后续管理请求都会携带 token。

adminService 对需要权限的操作执行 requireAdmin 校验，只有 status 为 enabled 且 role 为 super_admin 的管理员可以继续访问。数据看板接口统计 users、products、categories、chat_sessions 和 chat_messages 等集合数量。用户管理接口支持按状态和关键词查询用户，并可更新用户 status。商品管理接口支持查询商品和更新商品状态。分类管理接口支持读取和修改分类。聊天记录接口支持查看会话列表和消息明细。

禁用用户的完整逻辑不是只在后台按钮完成，而是由后台写状态、云函数读状态、小程序端处理错误三部分组成。后台将 users.status 更新为 disabled 后，小程序侧 login 云函数会拒绝该用户再次登录，user、product、favorite 和 chatService 云函数也会拒绝该用户继续访问已登录操作，从而形成真正的后端权限控制。

## 第4章 数据库设计及实现

### 4.1 数据库总体设计

系统使用微信云数据库作为主要数据存储。根据业务边界划分，主要集合包括 users、categories、products、favorites、chat_sessions、chat_messages 和 admins。users 保存小程序用户资料，categories 保存商品分类，products 保存商品信息，favorites 保存用户收藏关系，chat_sessions 保存聊天会话，chat_messages 保存聊天消息，admins 保存后台管理员信息。

数据库设计遵循三个原则。第一，核心实体独立成集合，例如用户、商品和会话分别存储，便于单独查询和维护。第二，部分常用展示信息采用快照冗余，例如商品中保存 sellerName 和 sellerAvatar，会话中保存 productSnapshot、buyerSnapshot 和 sellerSnapshot，以减少列表页频繁关联查询。第三，涉及状态流转的实体均设置 status 字段，例如用户 status 表示 enabled 或 disabled，商品 status 表示 on_sale、off_shelf 或 sold，会话 status 表示 active。

### 4.2 users 集合设计

users 集合用于保存小程序用户信息。推荐字段如下：

```json
{
  "_id": "数据库自动生成",
  "openid": "微信用户唯一标识",
  "account": "学生账号",
  "userId": "系统内部用户ID",
  "avatar": "头像云存储FileID",
  "nickname": "用户昵称",
  "bio": "个人简介",
  "gender": "性别，默认未知",
  "school": "学校名称",
  "status": "enabled 或 disabled",
  "verified": false,
  "createdAt": "创建时间",
  "updatedAt": "更新时间",

```

openid 和 account 均应保持唯一。status 字段是后台禁用用户的关键字段，默认 enabled。当管理员禁用用户时，该字段更新为 disabled。登录云函数和其他需要登录的云函数必须检查该字段，不能只依赖前端控制。

### 4.3 categories 集合设计

categories 集合用于保存商品分类。每条分类记录推荐字段如下：

```json

  "categoryId": "digital",
  "name": "数码电子",
  "icon": "📱",
  "description": "手机、电脑、平板、耳机等数码产品",

  "status": "enabled",
```

categoryId 用于程序内部识别，name 用于页面展示，icon 用于增强分类视觉识别，description 用于说明分类范围，sort 用于控制展示顺序，status 用于控制分类是否启用，productCount 可用于统计分类下商品数量。当前系统即使 categories 集合为空，前端也会提供五个默认分类兜底，保证页面可用。

## 第5章 系统的详细实现

### 5.1 项目实现采用的技术

小程序端采用微信小程序原生技术栈，包括 WXML、WXSS、JavaScript、微信小程序组件和微信云开发 SDK。WXML 负责页面结构，WXSS 负责样式，JavaScript 负责页面状态、事件处理、接口调用和数据渲染。小程序通过 wx.cloud.callFunction 调用云函数，通过 wx.cloud.uploadFile 上传头像和商品图片。

后端采用微信云函数，运行环境为 Node.js，主要依赖 wx-server-sdk。云函数按业务模块拆分，包括 login、user、category、product、favorite、chatService 和 adminService。每个云函数内部根据 action 参数执行具体操作，并统一返回 `{ code, message, data }` 格式，便于前端统一处理成功和失败结果。

数据层采用微信云数据库和云存储。云数据库保存结构化业务数据，云存储保存头像和商品图片。头像存储在 avatar 文件夹下，商品图片按发布模块的上传逻辑保存到云存储指定路径。前端页面拿到 fileID 后，将 fileID 写入数据库字段，页面展示时再通过 fileID 加载图片资源。

后台管理端采用 Vite 和原生 JavaScript 开发。后台页面由 admin-web/src/main.js 渲染，样式由 admin-web/src/styles.css 管理，接口调用由 admin-web/src/cloud.js 封装，登录状态由 admin-web/src/state.js 管理。后台通过 HTTP 访问 adminService 云函数，解决 Web 端直接访问微信云数据库权限和实时性的问题。

### 5.3 各个功能模块的实现

#### 5.3.1 登录模块实现

登录页面提交账号和密码后调用接口封装中的 login 方法：

```js
login: (data) => callFunction('login', { action: 'login', data })
```

login 云函数首先通过 `cloud.getWXContext().OPENID` 获取当前微信用户 openid，然后校验账号和密码。核心逻辑包括账号格式校验、固定口令校验、openid 查询、账号唯一性检查、用户初始化和禁用状态检查。登录成功后返回标准用户对象，前端将用户信息保存到本地状态并跳转首页。

登录模块的关键设计在于 openid 和账号双重唯一。openid 用于识别微信用户，account 用于识别学校账号。一个 openid 不能绑定多个账号，一个账号也不能被多个 openid 绑定。这样可以避免用户重复注册或冒用他人账号。

#### 5.3.2 用户资料模块实现

用户资料模块由 user 云函数和设置页面共同实现。设置页面负责选择头像、输入昵称、选择性别和填写个人简介。头像上传成功后得到 fileID，再调用 updateUserInfo 或 updateProfile 更新数据库。user 云函数只允许当前 openid 对应用户修改自己的资料，并会检查用户是否存在、是否被禁用。

用户资料在“我的”页面展示。头像区域展示用户头像、昵称、学校名称、认证状态和个人简介。个人简介采用更具辨识度的字体样式，使其与基础资料区分。学校图标、学校名和认证状态作为一组信息展示在昵称右侧，提高头像区域的信息密度。

#### 5.3.3 分类模块实现

分类模块由 category 云函数、首页分类区域和发布页分类选择器共同组成。category 云函数读取 categories 集合中 status 为 enabled 的分类，并按 sort 排序返回。前端保留默认分类数组作为兜底，避免云函数返回空数组或 categories 集合尚未初始化时页面出现空白。

发布页选择分类时弹出选择器，分类选项始终包含五个基础分类。该设计解决了开发过程中出现的“弹出界面为空白”的问题，使发布流程不依赖数据库是否已完全初始化。

#### 5.3.4 商品模块实现

商品模块是系统核心模块之一。发布商品时，前端调用 createProduct，云函数校验用户和商品数据后写入 products 集合。商品列表通过 listProducts 读取，详情页通过 getProductDetail 读取，我的发布通过 getMyProducts 读取，商品状态通过 updateProductStatus 修改。

商品详情页不仅展示 products 集合中的商品字段，还会根据商品 openid 查询 users 集合中的卖家资料，以保证卖家头像和昵称使用最新值。这一处理解决了商品创建后卖家修改头像，详情页仍显示旧头像或根据昵称生成头像的问题。

#### 5.3.5 收藏模块实现

收藏模块由 favorite 云函数和详情页收藏按钮实现。用户进入商品详情页时调用 checkFavorite，点击按钮时根据当前状态调用 addFavorite 或 removeFavorite。favorite 云函数会校验用户状态，防止被禁用用户继续收藏商品。收藏成功后，商品 favoriteCount 同步增加；取消收藏后，favoriteCount 同步减少。

“我的收藏”页面调用 listFavorites 获取收藏商品列表。“我的”页面展示收藏量时，可以通过 getDashboard 等用户接口统计收藏数量，保证用户能够直观看到自己的收藏行为结果。

#### 5.3.6 消息模块实现

消息模块通过 chatService 云函数统一处理。openSession 负责打开或创建会话，sendMessage 负责发送消息，getMessages 负责获取消息列表，getSessionList 负责获取会话列表，markRead 负责标记已读。前端页面只需要围绕这些接口组织交互，不直接操作 chat_sessions 或 chat_messages 集合。

发送消息时，云函数先新增 chat_messages 记录，再更新 chat_sessions 的 lastMessage、lastMessageAt 和 unreadCount。lastMessage 采用整体对象写入，避免数据库在 lastMessage 为 null 时无法创建子字段的问题。发送成功后，前端清空输入框并刷新消息列表。

#### 5.3.7 后台管理模块实现

后台管理端位于 admin-web 目录。管理员登录后进入主布局，左侧是导航菜单，右侧是当前功能页面。后台包括数据看板、用户列表、商品列表、分类列表和聊天记录五个主要视图。所有后台请求通过 callAdmin 方法发送到 adminService 云函数 HTTP 地址，adminService 根据 action 分发到具体处理函数。

后台用户管理中，禁用用户按钮会调用 updateUserStatus，将 users.status 更新为 disabled。后端云函数随后在登录、用户资料、商品、收藏和聊天接口中读取该状态并拦截。这样后台管理操作不只是页面上的状态变化，而是真正影响小程序端用户权限。

### 5.4 关键技术和算法实现

本项目没有复杂的机器学习或高级算法，关键技术主要体现在权限控制、数据一致性和推荐策略上。

第一，用户禁用采用后端强校验。只在前端隐藏入口是不可靠的，因为用户仍可能通过已有登录态调用云函数。因此系统在 login、user、product、favorite 和 chatService 中统一检查 users.status。该设计确保禁用用户无法登录，也无法继续发布商品、收藏商品或发送消息。

第二，收藏数量和会话摘要采用冗余更新。favorites 保存收藏关系，products.favoriteCount 保存商品收藏数量；chat_messages 保存消息明细，chat_sessions.lastMessage 保存最后一条消息摘要。这种设计提高了列表页读取效率，但要求写入时同步更新。系统通过云函数集中处理写逻辑，减少前端直接操作多集合导致的数据不一致风险。

第三，首页推荐采用轻量排序策略。当前阶段不引入复杂推荐算法，而是优先展示在售商品，并按创建时间或更新时间倒序排列。这样可以保证首页持续有内容展示，也能让新发布商品获得曝光。后续如果需要个性化推荐，可以在此基础上加入用户收藏分类、浏览记录、搜索关键词和商品热度等因素。

第四，后台 Web 端通过云函数 HTTP 化访问云数据库。由于小程序数据存放在微信云开发环境中，普通 Web 前端不能直接安全访问云数据库。因此后台管理端不直接连接数据库，而是访问 adminService，由云函数在服务端完成身份验证和数据库操作。这种方式更符合最小权限原则。

## 第6章 系统测试

### 6.1 测试目标

系统测试的目标是验证各功能模块是否满足需求，验证前端页面与云函数接口是否能够正确对接，验证数据库字段是否按预期写入和读取，验证异常场景是否能够返回明确提示。由于项目采用小程序端、云函数端和后台 Web 端组合结构，测试需要覆盖用户端流程、后台管理流程和数据联动流程。

### 6.2 登录模块测试

登录测试包括合法账号登录、非法账号登录、错误密码登录、首次登录初始化、非首次登录读取资料和禁用用户登录。合法账号示例为 222023603052106，密码为 swu123456。首次登录后，应在 users 集合生成用户记录，头像为系统默认头像，昵称为“用户+6位随机数字”，性别为“未知”，状态为 enabled。再次登录时，应读取已有头像、昵称、简介和性别，不应覆盖用户已修改资料。

非法账号包括非数字账号、不是 22 开头的账号、长度不是 15 位的账号。错误密码包括空密码和非默认口令。后台禁用用户后，再使用该账号登录，应提示账号已被禁用，并拒绝进入小程序。

### 6.3 用户资料测试

用户资料测试包括头像更新、昵称更新、性别更新、个人简介更新和空字段处理。头像更新时，前端应先上传云存储，再将 fileID 写入 users.avatar。昵称和简介应以字符串保存，性别可选择男、女或未知。测试过程中曾发现设置页面修改性别后提示“没有可更新的资料”，原因是更新逻辑未正确识别 gender 字段变化，修复后性别字段可以正常提交。

### 6.4 商品模块测试

商品模块测试包括分类加载、商品发布、图片上传、首页推荐、搜索、详情页展示、我的发布和商品状态修改。发布页面在 categories 集合为空时仍应显示五个默认分类，避免用户无法选择分类。商品发布成功后，products 集合应新增记录，状态为 on_sale，图片字段保存云存储 fileID。

首页推荐测试要求在未搜索状态下也能看到已发布的在售商品。搜索测试要求输入关键词后展示匹配商品，点击取消按钮后回到首页推荐状态。详情页测试要求展示商品标题、价格、分类、位置、描述、图片和卖家信息。曾出现点击商品进入详情页为空白的问题，修复后详情页能够正常渲染商品数据。

我的发布测试要求页面只保留在售、已下架和已售出三个板块，不显示“全部”板块，也不显示编辑按钮。状态修改测试要求卖家可以将商品下架或标记已售出，修改后列表分类正确变化。

### 6.5 收藏模块测试

收藏模块测试包括收藏商品、取消收藏、收藏状态回显、我的收藏列表和收藏数量统计。用户在详情页点击收藏后，favorites 集合应新增记录，商品 favoriteCount 增加，按钮状态变为已收藏。再次点击取消收藏后，收藏记录应取消，favoriteCount 减少。我的收藏页面应展示已收藏商品，“我的”页面收藏量应随收藏数量变化。

测试过程中曾发现详情页点击收藏没有反应，原因与 favorite 云函数部署和前端调用有关。上传云函数并修复统计逻辑后，收藏列表和收藏数量能够正常展示。

### 6.6 消息模块测试

消息模块测试包括发起会话、发送消息、消息列表刷新、输入框清空、会话 lastMessage 更新和未读数更新。用户只能对在售和已售出商品发起新会话，已下架商品不允许新建会话。发送消息后，chat_messages 应新增消息记录，chat_sessions.lastMessage 应更新为对象结构，unreadCount 应按接收方增加。

测试过程中曾出现发送消息前端提示失败但稍后消息成功发送的问题，说明前端响应处理和后端更新会话存在不一致。之后又出现 lastMessage 为 null 时更新 lastMessage.content 报错的问题，错误信息为不能在 null 元素中创建 content 字段。修复方式是发送消息时整体写入 lastMessage 对象，而不是对子字段进行更新。修复后消息发送、输入框清空和会话摘要更新均正常。

### 6.7 后台管理测试

后台管理测试包括管理员初始化、管理员登录、数据看板加载、用户列表加载、商品列表加载、分类列表加载、聊天记录查看、禁用用户和商品状态修改。管理员账号为 wu 和 xiao，初始密码为 campusx123456。首次使用需要调用 initAdmins 初始化 admins 集合。登录成功后应获得 token，后续请求携带 token。

后台禁用用户后，需要到小程序端验证禁用效果。测试发现仅后台写入 users.status 并不能自动阻止小程序登录，必须在 login 等云函数中增加状态检查。补充后，被禁用用户无法继续登录，且收藏、商品和聊天等已登录操作也会被拦截。

### 6.8 测试结论

经过阶段性测试，系统核心功能已经基本实现，包括用户登录、用户初始化、资料更新、分类展示、商品发布、首页推荐、商品详情、收藏、我的发布、消息聊天、后台管理和禁用用户。当前仍需要后续补充的测试包括多用户并发聊天、云函数异常日志分析、数据库索引优化、后台正式部署后的跨域访问测试以及协议页面内容补全。

## 第7章 总结与展望

### 7.1 总结

CampusX 校园二手交易小程序围绕校园交易场景，完成了从用户端到后台管理端的基础闭环。小程序端提供登录、首页推荐、分类浏览、商品发布、商品详情、收藏、消息、个人中心、设置、我的发布和关于我们等页面；云函数端按照业务模块拆分，实现登录初始化、用户资料、分类、商品、收藏、聊天和后台管理等能力；云数据库端设计了 users、categories、products、favorites、chat_sessions、chat_messages 和 admins 等集合；后台 Web 端提供管理员登录、数据看板、用户管理、商品管理、分类管理和聊天记录查看。

项目开发过程中采取了逐模块推进的方式。先完成登录模块和用户初始化，再完成商品分类与发布，然后补充首页推荐、收藏、商品状态管理和消息聊天，最后设计后台管理系统。这样的开发节奏降低了一次性设计全部云函数带来的不确定性，也便于在每个阶段完成测试和修复。

从实现结果看，系统已经具备校园二手交易平台的主要能力。用户可以发布和浏览商品，围绕商品进行沟通，收藏感兴趣的商品，并维护个人资料。管理员可以通过后台掌握平台运行情况，并对用户和商品进行基础治理。系统中一些关键问题也在迭代中得到解决，例如分类为空时页面空白、详情页渲染异常、收藏数量不更新、聊天消息发送失败、lastMessage 更新错误和禁用用户仍可登录等。

### 7.2 展望

后续可以从功能完善、体验优化、安全增强和运营能力四个方向继续扩展。

功能完善方面，可以补充订单模块，使买家和卖家的交易从“沟通”进一步发展到“确认购买、交易完成、评价反馈”的闭环。当前系统已经预留已售出状态，但买家侧“已购买”历史尚未实现，后续可以设计 orders 集合，记录买家、卖家、商品、交易状态和评价信息。

体验优化方面，可以进一步完善推荐策略。当前首页推荐采用在售商品按时间排序的轻量方案，后续可以根据用户浏览记录、收藏分类、搜索关键词、商品热度和发布时间综合排序，提高推荐准确性。也可以加入价格筛选、校区筛选、成色筛选和排序切换等功能。

安全增强方面，可以继续完善账号体系和后台权限体系。当前用户登录第一版采用固定口令校验，后续可以接入学校统一身份认证或更安全的验证码机制。后台管理员当前只有超级管理员角色，后续可以扩展普通管理员、审核员等角色，并对用户管理、商品管理、分类管理和聊天查看设置不同权限。

运营能力方面，可以增加商品审核、举报处理、敏感词过滤、系统通知和数据报表导出。对于校园平台而言，违规商品、虚假信息和不当聊天内容都需要管理机制。后台可以增加审核队列、举报列表和处理记录，使平台治理更加规范。

部署和运维方面，后台 Web 端当前先以本地开发方式运行，后续可以部署到静态网站托管或学校服务器，并配置正式域名、HTTPS 和访问控制。云数据库也可以根据查询情况添加索引，减少商品列表、聊天列表和后台统计的查询成本。

总体而言，CampusX 已经完成一个校园二手交易系统的主要架构和核心功能，为后续课程展示、实际试用和持续迭代提供了较完整的基础。
